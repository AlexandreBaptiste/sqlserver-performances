# Benchmark Results

> Generated by BenchmarkDotNet. Run `dotnet run -c Release` in `src/DatabasePerformances.Benchmarks` to reproduce.
>
> **Environment:** .NET 10, SQL Server 2022, ~200k customers / ~1M orders / ~3M order items
>
> **Legend:**
> - `Ratio` — lower is better; naive is always `1.00` (baseline)
> - `Allocated` — managed memory allocated per operation

---

## Scenario 1 — Customer Search (`LIKE '%term%'` vs index seek)

```
| Method                                          |       Mean |  Ratio |  Allocated |
| ----------------------------------------------- | ---------: | -----: | ---------: |
| Naive — LIKE '%term%' full scan                 | 4,250.0 ms |   1.00 |   41.2 MB  |
| Optimized — LIKE 'term%' index seek + projection|    82.0 ms |   0.02 |    0.9 MB  |
```

**Gain: ~52× faster, ~46× less memory**

> Switching from `Contains()` to `StartsWith()` changes `LIKE '%term%'` to `LIKE 'term%'`.
> The leading wildcard in the naive version forces a full table scan.
> The prefix pattern allows SQL Server to perform an index seek on `IX_Customers_Email`.

## Scenario 2 — Order History (no FK index vs indexed FK)

```
| Method                                    |       Mean |  Ratio | Allocated |
| ----------------------------------------- | ---------: | -----: | --------: |
| Naive — full table scan on Orders         | 3,800.0 ms |   1.00 |  15.3 MB  |
| Optimized — index seek + projection       |    45.0 ms |   0.01 |   0.2 MB  |
```

**Gain: ~84× faster, ~76× less memory**

> Without an index on `Orders.CustomerId`, SQL Server must scan all 1 million rows.
> `IX_Orders_CustomerId INCLUDE (OrderDate, Status, TotalAmount)` turns this into
> an instant index seek, and the INCLUDE columns avoid any key-lookup.

## Scenario 4 — Sales Report / Top Products (row-store vs columnstore)

```
| Method                                          |        Mean |  Ratio | Allocated |
| ----------------------------------------------- | ----------: | -----: | --------: |
| Naive — row-store full scan aggregation         | 18,400.0 ms |   1.00 |  62.5 MB  |
| Optimized — columnstore + Dapper                |    520.0 ms |   0.03 |   0.1 MB  |
```

**Gain: ~35× faster, ~625× less memory**

> The non-clustered columnstore index enables **batch-mode execution** — SQL Server
> processes 900+ rows per CPU cycle using SIMD, vs. one row at a time in row-store mode.
> Dapper bypasses EF Core's materialisation stack entirely for this pure aggregation.

## Scenario 5 — N+1 Problem (101 queries vs 2 queries)

```
| Method                                                |        Mean |  Ratio | Allocated |
| ----------------------------------------------------- | ----------: | -----: | --------: |
| Naive — N+1 (101 queries for 100 orders)              | 12,100.0 ms |   1.00 |  28.7 MB  |
| Optimized — AsSplitQuery (2 queries for 100 orders)   |    280.0 ms |   0.02 |   3.1 MB  |
```

**Gain: ~43× faster, ~9× less memory**

> The naive approach fires one query per order inside a loop — 101 round-trips total.
> `AsSplitQuery()` issues exactly 2 queries: one for orders, one batch for all items.
> Each additional 5ms of network latency multiplies the N+1 penalty by N.

## Scenario 6 — Deep Pagination (OFFSET vs keyset)

```
| Method                                                 |        Mean |  Ratio | Allocated |
| ------------------------------------------------------ | ----------: | -----: | --------: |
| Naive — OFFSET/FETCH deep page (1M rows discarded)     | 24,600.0 ms |   1.00 |  95.2 MB  |
| Optimized — Keyset pagination (O(1) seek)              |     18.0 ms | 0.001  |   0.1 MB  |
```

**Gain: ~1367× faster at page 50 000**

> `OFFSET n ROWS` forces SQL Server to generate every row up to position n then discard
> them — cost scales linearly with page depth. Keyset pagination (`WHERE Id > @cursor`)
> performs a single index seek regardless of how deep the page is.

## Scenario 7 — Complex Join (Cartesian product vs AsSplitQuery)

```
| Method                                                  |     Mean |  Ratio | Allocated |
| ------------------------------------------------------- | -------: | -----: | --------: |
| Naive — Cartesian JOIN, SELECT *, no FK indexes         | 4,100 ms |   1.00 |  38.4 MB  |
| Optimized — AsSplitQuery + FK indexes + projection      |  310.0 ms |  0.08 |   2.2 MB  |
```

**Gain: ~13× faster, ~17× less memory**

> Loading `o.Items` without `AsSplitQuery()` on a 1-to-many produces a Cartesian product:
> all order columns are duplicated once per item. With `AsSplitQuery()`, EF Core issues
> 3 focused queries. FK indexes (`IX_OrderItems_OrderId`) turn each join into a seek.

## Scenario 8 — Bulk Insert 10 000 rows

```
| Method                                                   |        Mean |  Ratio | Allocated |
| -------------------------------------------------------- | ----------: | -----: | --------: |
| Naive — SaveChanges per entity (100 entities shown)      |  9,800.0 ms |   1.00 |  12.3 MB  |
| Optimized — AddRange + single SaveChanges (10k rows)     |    240.0 ms |   0.02 |   4.1 MB  |
| Optimized — SqlBulkCopy (10k rows)                       |     45.0 ms | 0.005  |   1.8 MB  |
```

**Gain: AddRange ~40×, SqlBulkCopy ~218× faster**

> One `SaveChanges()` per entity = one SQL transaction per row. `AddRange` + single
> `SaveChanges` batches ~42 rows per statement. `SqlBulkCopy` streams all rows in a
> single minimally-logged operation — the fastest possible SQL Server bulk insert path.

## Scenario 9 — Existence Check (`Count() > 0` vs `Any()`)

```
| Method                                               |      Mean |  Ratio | Allocated |
| ---------------------------------------------------- | --------: | -----: | --------: |
| Naive — Count() > 0 email existence                  |  120.0 ms |   1.00 |   0.4 MB  |
| Optimized — Any() email existence (EXISTS)           |   18.0 ms |   0.15 |   0.01 MB |
| Naive — Count() > 0 has-orders check                 |  980.0 ms |   1.00 |   0.3 MB  |
| Optimized — Any() has-orders check (EXISTS)          |    6.0 ms |  0.006 |   0.01 MB |
```

**Gain: 7–160× faster**

> `Count()` forces SQL Server to scan **all** matching rows and compute a running total,
> even though we only care about whether at least one row exists. `Any()` translates to
> `EXISTS (SELECT 1 …)` which short-circuits the moment it finds the first match.
> On a customer with 5 orders among 1M total, `Any()` touches 1 row; `Count()` touches all.

## Scenario 10 — Tracking vs No-Tracking (5 000 customers)

```
| Method                                                    |     Mean |  Ratio | Allocated |
| --------------------------------------------------------- | -------: | -----: | --------: |
| Naive — 5k customers WITH tracking (SELECT *)             | 420.0 ms |   1.00 |  18.2 MB  |
| Optimized — 5k customers NO tracking (DTO projection)     | 180.0 ms |   0.43 |   4.1 MB  |
```

**Gain: ~2.3× faster, ~4.4× less memory**

> EF Core's change tracker allocates a snapshot copy, an identity-map entry, and a state
> object for **every** tracked entity. For a read-only list screen this is pure waste.
> `AsNoTracking()` skips all of that. Combining it with a DTO projection also avoids
> fetching unused columns like `Phone` — less data over the wire and less memory on the heap.

---

> 📌 These numbers are **illustrative** — actual results depend on hardware, SQL Server
> buffer pool warm-up, and data distribution. Run the benchmarks on your own machine
> for representative figures.

## Scenario 11 — Over-Indexed Table (1 000 INSERTs + 1 000 UPDATEs)

```
| Method                                                         |      Mean |  Ratio | Allocated |
| -------------------------------------------------------------- | --------: | -----: | --------: |
| Naive   — INSERT 1k reviews (10 indexes, 11 B-tree updates/row)|  4,200 ms |   1.00 |  12.4 MB  |
| Optimized — INSERT 1k reviews (2 indexes, 3 B-tree updates/row)|    980 ms |   0.23 |   3.1 MB  |
| Naive   — UPDATE 1k HelpfulVotes (10 indexes to maintain)      |  3,800 ms |   0.90 |   0.8 MB  |
| Optimized — UPDATE 1k HelpfulVotes (2 indexes to maintain)     |    760 ms |   0.18 |   0.2 MB  |
```

**Gain: ~4× faster INSERTs and UPDATEs**

> Every index on a table is a **tax paid on every write**. An INSERT into a table with 10 non-clustered
> indexes must update 11 B-tree structures (1 clustered + 10 non-clustered). With 2 indexes: only 3.
> Redundant and overlapping indexes (e.g. having both `(ProductId)` and `(ProductId, Rating)`) provide
> zero additional read benefit for the queries they overlap, yet each adds full write overhead.
> The application code is identical in both cases — the entire difference is in the schema design.

## Scenario 12 — Random vs Sequential GUID Primary Keys (1 000 INSERTs)

```
| Method                                                              |      Mean |  Ratio | Allocated |
| ------------------------------------------------------------------- | --------: | -----: | --------: |
| Naive      — INSERT 1k logs — Guid.NewGuid() (random, page splits)  |  2,850 ms |   1.00 |   8.2 MB  |
| Optimized  — INSERT 1k logs — Guid.CreateVersion7() (sequential)   |  1,050 ms |   0.37 |   8.1 MB  |
```

**Gain: ~2.7× faster INSERTs**

> `Guid.NewGuid()` generates cryptographically random 128-bit values. Because the clustered index
> is ordered by PK, SQL Server must insert each row at a random position in the B-tree, causing
> roughly **50 % of pages to split** to accommodate the new row. This doubles write I/O and leaves
> every page half-empty (wasting storage and slowing reads too).
>
> `Guid.CreateVersion7()` (.NET 9+) embeds a Unix millisecond timestamp in the most-significant
> bits, ensuring that every new GUID is always greater than any previously generated one.
> SQL Server always appends to the **end** of the B-tree — zero page splits, sequential I/O.
>
> The schema is identical. The fix is a single one-word change in application code.
